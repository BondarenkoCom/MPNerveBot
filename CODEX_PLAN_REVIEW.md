# Codex Review of PLAN.md

Date: 2026-03-09
Author: Codex
Scope: review of `PLAN.md` before MVP implementation

---

## Что я поняла

- Делаем не SaaS и не веб-кабинет, а Telegram-first operational bot для продавцов Wildberries и Ozon.
- MVP ядро: онбординг с токенами, ежедневный дайджест, алерты каждые 2 часа, работа с отзывами, AI-черновики ответов.
- Базовый стек выбран адекватно для быстрого MVP: Python 3.11+, `python-telegram-bot` async, `httpx`, `SQLite`, mock-first подход.
- До первых реальных токенов проект должен стабильно жить через `mock_api`, причем интерфейсы mock/real клиентов должны быть взаимозаменяемыми.
- Все user-facing сообщения бота должны быть на русском.
- Рабочая зона ограничена `D:\Projects\Repos\MPNerveBot\`. Никаких действий вне проекта.

## Что в плане сильное

- Узкий фокус по продукту, без расползания в generic SaaS.
- Есть понятный execution order по фазам.
- Правильно заложен mock-first подход до пилотного продавца.
- Есть разделение ролей между Artem, Sonnet и Codex.

## Что в плане пока недоговорено

- Нет точной архитектуры слоев и контрактов между `handlers -> services -> models -> scheduler`.
- Нет схемы БД и явной модели данных для seller, token, alert settings, reviews, digest history, job state.
- Нет формального onboarding flow: шаги, состояния, валидация токенов, обработка ошибок и повторный ввод.
- Нет integration plan по WB/Ozon: какие именно методы нужны в MVP, как устроить проверку токена, как строить graceful fallback на mock.
- Нет композиции приложения: где выбирается real/mock client, как scheduler получает доступ к зависимостям, где хранится runtime config.
- Нет явной стратегии тестирования для mock data, digest formatting, alert logic и onboarding flows.

## Мое мнение

Текущий план годится как продуктовый master plan, но еще не годится как технический источник истины для параллельной разработки двумя агентами.

Если пойти сразу в код, будет почти гарантированная рассинхронизация:
- Sonnet может описать одни data shapes для mock/API.
- Codex может зашить другие контракты в handlers и scheduler.
- Потом начнется стыковка через переделки, а не через заранее согласованные интерфейсы.

Слабое место не в стеке и не в фичах. Слабое место в отсутствии технических контрактов между модулями.

## Что я предлагаю сделать до активной кодовой фазы

1. Зафиксировать MVP architecture note:
   `app composition`, границы слоев, зависимости, lifecycle бота и scheduler.
2. Зафиксировать data model:
   таблицы SQLite, ключевые поля, статусы онбординга, сущности marketplace account / alert config / review item / digest snapshot.
3. Зафиксировать integration plan:
   какие методы нужны от `wb_api.py`, `ozon_api.py`, `mock_api.py`, и какой унифицированный формат ответа возвращает каждый клиент.
4. Зафиксировать onboarding flow:
   happy path и error paths для `WB`, `Ozon`, `оба`, invalid token, skip/redo.
5. Только после этого собирать каркас проекта и писать код.

## Что я буду делать дальше

1. Проведу аудит текущего состояния репозитория и сверю, что уже создано, а что пока только в плане.
2. Подготовлю архитектурный foundation для MVP:
   roadmap, data model, onboarding flow, API integration plan, composition rules.
3. Затем начну реализацию каркаса проекта и первых рабочих файлов.
4. После появления мнения Sonnet сравню оба разбора и соберу согласованный итоговый technical direction.

## Ожидаемое участие Sonnet

От Sonnet нужен не пересказ плана, а независимая критика по трем точкам:
- достаточно ли плана для параллельной работы двух агентов;
- какие контракты нужно зафиксировать первыми;
- какие части MVP можно реально писать сразу, а какие нельзя трогать без доп. спецификации.

## Предварительный вывод

Проект понятный и жизнеспособный.
Но прямо сейчас главный риск не в API Wildberries/Ozon и не в Telegram framework.
Главный риск - начать кодить без предварительно зафиксированных контрактов между слоями и агентами.
